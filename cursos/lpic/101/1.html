101.1 Determinar y configurar los ajustes de hardware

El administrador de sistemas debe tener visibilidad total sobre los componentes físicos y virtuales. Más allá de conocerlos, debemos saber cómo interactuar con ellos a través del kernel.

Herramientas de diagnóstico de hardware

Además de los comandos básicos, existen utilidades específicas para obtener información profunda (DMI) y eventos en tiempo real.

  • lspci / lsusb:
    Mapeo de buses PCI y USB. Usar -v (verbose) y -k (kernel driver) es esencial para troubleshooting.
  • dmidecode:
    Herramienta potente que lee la tabla DMI (SMBIOS) de la placa base. Muestra número de serie, versión de BIOS, tipo de RAM y velocidad de los procesadores sin necesidad de abrir el chasis. Requiere root. 
    sudo dmidecode -t memory  # Info detallada de memoria
sudo dmidecode -s system-serial-number
  • udevadm:
    Interfaz para gestionar eventos del dispositivo udev. 
    udevadm info -a -p /sys/class/net/eth0  # Atributos de una NIC
udevadm monitor                       # Ver eventos de conexión en tiempo real
  • hdparm / sdparm:
    Permiten obtener y establecer parámetros de discos duros (SATA/IDE/SCSI), como caché de escritura o gestión de energía. 
    hdparm -I /dev/sda    # Info detallada del disco
hdparm -tT /dev/sda   # Test de velocidad de lectura

Gestión de Módulos del Kernel

El kernel Linux es modular. Podemos cargar o descargar controladores en caliente.

  • lsmod: Lista módulos cargados. Muestra el tamaño y qué otros módulos dependen de él (Used by).
  • modinfo nombre_modulo: Muestra información detallada, autor, parámetros y dependencias.
  • modprobe: La herramienta estándar para cargar/descargar. Resuelve dependencias automáticamente. 
    modprobe -r vmwgfx    # Descargar módulo
modprobe vmwgfx        # Cargar módulo
  • Configuración: Los parámetros de los módulos se pueden pasar en /etc/modprobe.d/nombre.conf. Para bloquear un módulo problemático: blacklist modulo_malo.

Sistemas de archivos /proc, /sys y /dev

Linux expone el hardware como archivos. Entender esta estructura es clave.

/proc
Ventana al Kernel
cpuinfo, interrupts, meminfo, dma
/sys
Exportación de objetos kobject
Estructura dinámica de dispositivos.
/dev
Nodos de dispositivos
Gestionados por udev.

En /dev, los dispositivos de bloques (almacenamiento) tienen números mayor (tipo de dispositivo) y menor (instancia específica), visibles con ls -l.

101.2 Arranque del Sistema

Comprender la secuencia de arranque es vital para la recuperación de sistemas (boot repair).

BIOS vs UEFI

Característica BIOS (Legacy) UEFI
Partición de Arranque MBR (512 bytes, max 4 particiones primarias) GPT (Partición EFI System - ESP, formato FAT32)
Gestor de Arranque Cargado desde MBR (Etapa 1) Archivo .efi en ESP (/boot/efi/EFI/...)
Seguridad Básica Secure Boot (Firma de drivers/Bootloaders)

GRUB 2 (Grand Unified Boot Loader)

El archivo de configuración principal es /boot/grub/grub.cfg, pero no se debe editar manualmente, ya que se regenera.

  • Configuración: Editar /etc/default/grub.
  • Actualización: Ejecutar update-grub (Debian/Ubuntu) o grub2-mkconfig -o /boot/grub2/grub.cfg (RHEL/CentOS).

Parámetros críticos del kernel (Kernel Parameters) en GRUB_CMDLINE_LINUX:

root=/dev/mapper/vg_root-lv_root  # Ubicación del sistema raíz
ro                                # Montar raíz como Read-Only inicialmente
quiet splash                      # Ocultar output textual
init=/bin/bash                    # Reemplazar el init system (Útil para rescate)
systemd.unit=rescue.target        # Arrancar directamente en modo rescate

Kernel e Initramfs

El kernel (vmlinuz) se carga en RAM. Para acceder al disco real (que puede estar en LVM, RAID o cifrado), necesita un sistema de archivos temporal: initramfs (o initrd).

Una vez montado el sistema real (/), se ejecuta el PID 1. Históricamente /sbin/init, hoy en día /lib/systemd/systemd.

Diagnóstico del Arranque

  • dmesg: Muestra el buffer del anillo del kernel. dmesg -T muestra marcas de tiempo humanas.
  • journalctl: El registro de systemd. 
    journalctl -b               # Logs del arranque actual
journalctl -b -1             # Logs del arranque anterior
journalctl -p err -b         # Solo errores del arranque
  • systemd-analyze: Analiza el rendimiento del arranque. 
    systemd-analyze time                    # Tiempo total
systemd-analyze blame                   # Servicios que más tardan

101.3 Cambiar runlevels / targets y apagar el sistema

La gestión del "modo" en el que opera el sistema define qué servicios están activos.

SysVinit (Runlevels y /etc/inittab)

En sistemas heredados, /etc/inittab define el comportamiento. Su sintaxis es:

id:3:initdefault:

Donde id es el identificador, 3 es el runlevel por defecto, y initdefault es la acción.

0
Halt (Apagado)
1
Single User (Root sin red)
2
Multiusuario (sin NFS en Debian)
3
Multiusuario completo (Texto)
4
No usado (Personalizable)
5
Gráfico (X11)
6
Reboot

Los scripts de inicio residen en /etc/init.d/ y se enlazan en /etc/rcX.d/.

  • S##nombre: Start (Se ejecutan al entrar al runlevel con orden ##).
  • K##nombre: Kill (Se ejecutan al salir del runlevel para detener servicios).

Comando para recargar /etc/inittab sin reiniciar: telinit q.

Systemd (Targets)

Systemd abstrae los runlevels en "targets" (objetivos). Son unidades que agrupan otras unidades.

systemctl get-default               # Ver target actual
systemctl set-default graphical.target # Establecer por defecto
systemctl isolate multi-user.target    # Cambiar modo inmediatamente

Procedimientos de Recuperación (Rescue Mode)

Si olvidas la contraseña de root o el sistema no arranca, puedes intervenir en GRUB:

  1. En el menú de GRUB, presiona e para editar.
  2. Busca la línea que empieza con linux o linux16.
  3. Añade rd.break o init=/bin/bash al final de la línea.
  4. Presiona Ctrl+x para arrancar.
